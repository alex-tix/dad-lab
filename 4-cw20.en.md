## 1. Token Standards — Why They Exist

### The Problem: Composability

Imagine you build a decentralized exchange (DEX). You want it to support trading of *any* token — not just the ones you know about today, but also tokens that haven't been created yet. How do you write code that interacts with a token contract if you don't know what that contract looks like?

The answer is **standards**. A token standard is a shared interface — a contract that says "if you implement these exact messages and queries, any other contract or frontend can interact with you without knowing your internal details."

This is the same principle as USB. You don't need to know how a keyboard works internally — you just need to know it speaks the USB protocol, and any computer with a USB port can use it.

### ERC Standards (Ethereum)

Ethereum pioneered the idea of token standards through **ERCs** (Ethereum Request for Comments):

| Standard | Purpose | Key Example |
|----------|---------|-------------|
| ERC-20   | Fungible tokens (every token is identical) | USDC, LINK, UNI |
| ERC-721  | Non-fungible tokens (each token is unique) | CryptoPunks, Bored Apes |
| ERC-1155 | Multi-token (fungible + non-fungible in one contract) | Game items |

ERC-20 is the most fundamental. It defines functions like `transfer()`, `balanceOf()`, `approve()`, and `transferFrom()`. Because every fungible token on Ethereum implements ERC-20, Uniswap can list any token without custom integration — it just calls the standard functions.

### CW Standards (CosmWasm)

CosmWasm adopted the same idea, with naming that mirrors ERC:

| Standard | Purpose | ERC Equivalent |
|----------|---------|----------------|
| CW-20    | Fungible tokens | ERC-20 |
| CW-721   | Non-fungible tokens | ERC-721 |
| CW-1     | Proxy contracts | — |
| CW-3     | Multisig | — |
| CW-4     | Group membership | — |

The `cw-plus` repository is the canonical home for these specifications and their reference implementations.

### Key Differences from ERC-20

CW-20 is *inspired by* ERC-20 but differs in important ways:

1. **No reentrancy risk.** Remember the Actor Model from Lab 3 — contracts run to completion before sub-messages execute. The `approve` + `transferFrom` pattern in ERC-20 exists partly because Solidity's call semantics make `send`-style messages dangerous. CosmWasm doesn't have this problem.

2. **`Send` instead of `approve` + `transferFrom`.** CW-20 introduces `Send`, which transfers tokens to a contract *and* triggers an action on that contract in a single message. The receiving contract gets a `Receive` callback. This is more ergonomic and safer than ERC-20's two-transaction approval flow.

3. **`Burn` is a first-class action.** ERC-20 "burning" is usually implemented by transferring to a dead address (`0x000...`). CW-20 has an explicit `Burn` message that reduces the total supply.

4. **Decimals are cosmetic.** All amounts are stored as `Uint128` (128-bit unsigned integers). If your token has 6 decimals and someone holds "1.5 tokens", the chain stores `1500000`. The decimal count is purely metadata for UIs.

### Why This Matters for You

After this lab, any contract on Injective (a DEX, a lending protocol, a DAO treasury) will be able to interact with your token using the standard CW-20 messages. You won't need to explain your API to anyone — they already know it.

> **Injective note:** Injective also supports **TokenFactory** — native chain-level tokens managed by the `tokenfactory` module. For production use, Injective recommends TokenFactory due to its native integration with exchange modules. However, CW-20 is the universal CosmWasm standard, and Injective provides a **CW20 Adapter** contract to bridge between the two. Today we focus on CW-20 because the concepts apply across every CosmWasm chain.

## 2. The CW20 Specification

Before we write any code, let's map out what the CW-20 spec requires. The specification is split into **base** functionality (mandatory) and **extensions** (optional).

### Base (Required)

**Execute messages:**

| Message | What it does |
|---------|-------------|
| `Transfer { recipient, amount }` | Move tokens from sender to recipient (an address, not a contract) |
| `Burn { amount }` | Destroy tokens from the sender's balance, reducing total supply |
| `Send { contract, amount, msg }` | Transfer tokens to a contract *and* trigger an action on it |

**Queries:**

| Query | What it returns |
|-------|----------------|
| `Balance { address }` | The token balance of a given address |
| `TokenInfo {}` | Name, symbol, decimals, and total supply |

### Extensions (Optional)

| Extension | What it adds |
|-----------|-------------|
| **Mintable** | `Mint { recipient, amount }` — create new tokens (only callable by a designated minter) |
| **Allowances** | `IncreaseAllowance`, `DecreaseAllowance`, `TransferFrom`, `SendFrom`, `BurnFrom` — delegate spending rights |
| **Enumerable** | `AllAllowances`, `AllAccounts` — paginated list queries |
| **Marketing** | `UpdateMarketing`, `UploadLogo` — token metadata and logo |

### What We'll Build Today

We'll implement the **full base** plus the **mintable extension**. Specifically:

- `instantiate` — set token name/symbol/decimals, distribute initial balances, optionally set a minter
- `transfer`, `burn`, `send` — the three base execute messages
- `mint` — the mintable extension
- `balance`, `token_info` — the two base queries

Allowances and enumerable queries are left as exercises — they follow the same patterns you'll learn today.

## 4. State Design

A fungible token needs to track three things:

1. **Token metadata** — name, symbol, decimals, total supply, and optionally who can mint.
2. **Balances** — how many tokens each address holds.

Let's think about what storage primitives to use.

### Token Info (Singleton)

There is exactly one set of token metadata per contract instance. This maps perfectly to `Item<T>` — a single value stored under one key.

### Balances (Per-Address)

Every address that has ever held tokens needs its own balance entry. This maps to `Map<&Addr, Uint128>` — a key-value store where the key is an address and the value is the balance.

> **`Map` vs `Item<Vec<...>>`** — In Lab 3, we stored admins as `Item<Vec<Addr>>`. That works fine for small lists, but imagine a token with 10,000 holders. Loading the entire vector just to check one balance would be incredibly wasteful. `Map` stores each entry under its own key in the underlying KV store, so looking up a single balance is O(1).

Replace `src/state.rs`:

```rust
use cosmwasm_schema::cw_serde;
use cosmwasm_std::{Addr, Uint128};
use cw_storage_plus::{Item, Map};

#[cw_serde]
pub struct TokenInfo {
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
    pub total_supply: Uint128,
    pub mint: Option<MinterData>,
}

#[cw_serde]
pub struct MinterData {
    /// Address allowed to mint new tokens
    pub minter: Addr,
    /// Hard cap on total supply (None = unlimited minting)
    pub cap: Option<Uint128>,
}

pub const TOKEN_INFO: Item<TokenInfo> = Item::new("token_info");
pub const BALANCES: Map<&Addr, Uint128> = Map::new("balance");
```

**Why `MinterData` as a separate struct?** We need to store both *who* can mint and *how much* total can ever exist (the cap). Grouping these together keeps the data model clean. Making it `Option<MinterData>` means minting can be entirely disabled by passing `None` at instantiation.

## 5. Instantiation — Minting the Initial Supply

When someone instantiates our token contract, they need to provide:

- Token metadata (name, symbol, decimals)
- A list of initial balances (who gets how many tokens at genesis)
- Optionally, a minter configuration

### 5.1 Define the Messages

Replace `src/msg.rs`:

```rust
use cosmwasm_schema::{cw_serde, QueryResponses};
use cosmwasm_std::{Addr, Binary, Uint128};

// --- Instantiate ---

#[cw_serde]
pub struct InstantiateMsg {
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
    pub initial_balances: Vec<Cw20Coin>,
    pub mint: Option<MinterMsg>,
}

/// An initial balance assignment — address + amount.
#[cw_serde]
pub struct Cw20Coin {
    pub address: String,
    pub amount: Uint128,
}

/// Minter configuration in the instantiate message.
/// Uses `String` because we haven't validated the address yet.
#[cw_serde]
pub struct MinterMsg {
    pub minter: String,
    pub cap: Option<Uint128>,
}

// --- Query (placeholder, we'll expand this next) ---

#[cw_serde]
#[derive(QueryResponses)]
pub enum QueryMsg {}

// --- Responses ---

#[cw_serde]
pub struct TokenInfoResponse {
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
    pub total_supply: Uint128,
}

#[cw_serde]
pub struct BalanceResponse {
    pub balance: Uint128,
}
```

> **What is `#[cw_serde]`?** This macro from `cosmwasm-schema` is a shorthand that expands to all the derive macros a CosmWasm message type needs: `Serialize`, `Deserialize`, `Clone`, `Debug`, `PartialEq`, `JsonSchema`, plus the correct `#[serde(...)]` attributes. Using it avoids long, error-prone derive lists. For `QueryMsg`, we still need `#[derive(QueryResponses)]` separately because it's a different concern (mapping variants to return types).

### 5.2 Implement Instantiation

Replace `src/contract.rs`:

```rust
use cosmwasm_std::{
    to_json_binary, Addr, Binary, Deps, DepsMut, Env, MessageInfo, Response,
    StdError, StdResult, Uint128,
};

use crate::error::ContractError;
use crate::msg::{InstantiateMsg, QueryMsg};
use crate::state::{MinterData, TokenInfo, BALANCES, TOKEN_INFO};

pub fn instantiate(
    deps: DepsMut,
    _env: Env,
    _info: MessageInfo,
    msg: InstantiateMsg,
) -> Result<Response, ContractError> {
    // --- Validate token metadata ---
    if msg.name.is_empty() {
        return Err(ContractError::InvalidMetadata {
            reason: "Name must not be empty".to_string(),
        });
    }
    if msg.symbol.is_empty() {
        return Err(ContractError::InvalidMetadata {
            reason: "Symbol must not be empty".to_string(),
        });
    }
    if msg.decimals > 18 {
        return Err(ContractError::InvalidMetadata {
            reason: "Decimals must not exceed 18".to_string(),
        });
    }

    // --- Process initial balances ---
    let mut total_supply = Uint128::zero();

    for coin in &msg.initial_balances {
        let addr = deps.api.addr_validate(&coin.address)?;
        if coin.amount.is_zero() {
            continue; // Skip zero balances
        }
        BALANCES.save(deps.storage, &addr, &coin.amount)?;
        total_supply += coin.amount;
    }

    // --- Process minter config ---
    let mint = match msg.mint {
        Some(m) => {
            let minter = deps.api.addr_validate(&m.minter)?;
            // If a cap is set, it must be >= the initial supply
            if let Some(cap) = m.cap {
                if total_supply > cap {
                    return Err(ContractError::InvalidMetadata {
                        reason: "Initial supply exceeds mint cap".to_string(),
                    });
                }
            }
            Some(MinterData {
                minter,
                cap: m.cap,
            })
        }
        None => None,
    };

    // --- Save token info ---
    let token_info = TokenInfo {
        name: msg.name,
        symbol: msg.symbol,
        decimals: msg.decimals,
        total_supply,
        mint,
    };
    TOKEN_INFO.save(deps.storage, &token_info)?;

    Ok(Response::new()
        .add_attribute("action", "instantiate")
        .add_attribute("total_supply", total_supply.to_string()))
}

pub fn query(_deps: Deps, _env: Env, _msg: QueryMsg) -> StdResult<Binary> {
    unimplemented!()
}
```

**What just happened?** On instantiation, we: validated metadata, iterated over initial balances and saved each one to the `BALANCES` map, computed the total supply by summing them up, validated the minter config (if any), and persisted everything to storage. This is the token's "genesis" — from here on, only `execute` messages can change state.

## 6. Queries — Balance & Token Info

### 6.1 Add Query Variants

Update the `QueryMsg` enum and add response types in `src/msg.rs`. Replace the placeholder `QueryMsg`:

```rust
#[cw_serde]
#[derive(QueryResponses)]
pub enum QueryMsg {

    /// Returns token metadata. Returns TokenInfoResponse.
    #[returns(TokenInfoResponse)]

}
```

Also add the `MinterResponse` to the responses section of `src/msg.rs`:

```rust
#[cw_serde]

    pub minter: String,
    pub cap: Option<Uint128>,
```

### 6.2 Implement the Query Handlers

Add the query dispatch and handler module to `src/contract.rs`. Replace the `query` function:

```rust
pub fn query(deps: Deps, _env: Env, msg: QueryMsg) -> StdResult<Binary> {
    use QueryMsg::*;

    match msg {
        Balance { address } => to_json_binary(&query::balance(deps, address)?),
        TokenInfo {} => to_json_binary(&query::token_info(deps)?),
        Minter {} => to_json_binary(&query::minter(deps)?),
    }
}
```

And add the `query` module (add this at the end of `src/contract.rs`, before any `#[cfg(test)]` block):

```rust
mod query {
    use super::*;
    use crate::msg::{BalanceResponse, MinterResponse, TokenInfoResponse};

    pub fn balance(deps: Deps, address: String) -> StdResult<BalanceResponse> {
        let addr = deps.api.addr_validate(&address)?;
        let balance = BALANCES
            .may_load(deps.storage, &addr)?
            .unwrap_or_default();
        Ok(BalanceResponse { balance })
    }

    pub fn token_info(deps: Deps) -> StdResult<TokenInfoResponse> {
        let info = TOKEN_INFO.load(deps.storage)?;
        Ok(TokenInfoResponse {
            name: info.name,
            symbol: info.symbol,
            decimals: info.decimals,
            total_supply: info.total_supply,
        })
    }

    pub fn minter(deps: Deps) -> StdResult<MinterResponse> {
        let info = TOKEN_INFO.load(deps.storage)?;
        match info.mint {
            Some(m) => Ok(MinterResponse {
                minter: m.minter.to_string(),
                cap: m.cap,
            }),
            None => Err(StdError::generic_err("No minter configured")),
        }
    }
}
```

> **`may_load` vs `load`** — `BALANCES.load()` returns an error if the key doesn't exist. `may_load()` returns `Ok(None)` instead, which we default to zero. An address with no entry in the map simply has zero tokens — not an error.

```rust
                cap: Some(Uint128::new(10_000_000)),

        }

    fn proper_instantiation() {

        let msg = default_instantiate_msg(&app);
        let contract_addr = setup_contract(&mut app, msg);

        // Check token info
        let info: TokenInfoResponse = app
            .wrap()

            .unwrap();

        assert_eq!(info.name, "Test Token");
        assert_eq!(info.symbol, "TEST");
        assert_eq!(info.decimals, 6);
        assert_eq!(info.total_supply, Uint128::new(1_500_000));

        let msg = default_instantiate_msg(&app);
        let contract_addr = setup_contract(&mut app, msg);

        let alice = app.api().addr_make("alice");
        let nobody = app.api().addr_make("nobody");

        // Alice has 1,000,000
        let resp: BalanceResponse = app
            .wrap()
            .query_wasm_smart(
                &contract_addr,
                &QueryMsg::Balance {

        let resp: BalanceResponse = app
            .wrap()
            .query_wasm_smart(
                &contract_addr,
                &QueryMsg::Balance {
                    address: nobody.to_string(),
                },
            )
            .unwrap();
        assert_eq!(resp.balance, Uint128::zero());
    }

    #[test]
    fn query_minter() {
        let mut app = App::default();
        let msg = default_instantiate_msg(&app);
        let contract_addr = setup_contract(&mut app, msg);

        let minter_admin = app.api().addr_make("minter_admin");

        let resp: MinterResponse = app
            .wrap()
            .query_wasm_smart(&contract_addr, &QueryMsg::Minter {})
            .unwrap();

        assert_eq!(resp.minter, minter_admin.to_string());
        assert_eq!(resp.cap, Some(Uint128::new(10_000_000)));
    }

    #[test]
    fn cap_must_exceed_initial_supply() {
        let mut app = App::default();
        let code = ContractWrapper::new(execute, instantiate, query);
        let code_id = app.store_code(Box::new(code));

        let alice = app.api().addr_make("alice");
        let minter = app.api().addr_make("minter");
        let creator = app.api().addr_make("creator");

        let msg = InstantiateMsg {
            name: "Bad Token".to_string(),
            symbol: "BAD".to_string(),

                address: alice.to_string(),
                amount: Uint128::new(1_000_000),

                minter: minter.to_string(),

            }),

            .instantiate_contract(
                code_id,
                creator,
                &msg,
                &[],

```

> **Why `app.api().addr_make("alice")`?** In `cw-multi-test` 2.x, the mock API performs real bech32 validation when your contract calls `deps.api.addr_validate()`. Plain strings like `"alice"` are not valid bech32 addresses, and hardcoded `inj1...` addresses have the wrong prefix (the default mock uses `cosmwasm`). The `addr_make()` function generates a deterministic, valid bech32 address from any human-readable label — so `addr_make("alice")` always produces the same `cosmwasm1...` address. This means the same label gives you the same address across tests, which keeps assertions predictable.

## 8. Execute — Transfer

Now we add state-changing functionality. `Transfer` is the most fundamental: move tokens from the sender's balance to a recipient.

## 10. Execute — Mint

Minting creates new tokens out of thin air — but only the designated minter can do it, and the total supply must not exceed the cap (if one is set).

Replace the `mint` stub in the `exec` module:

```rust
    pub fn mint(
        deps: DepsMut,
        _env: Env,
        info: MessageInfo,
        recipient: String,
        amount: Uint128,
    ) -> Result<Response, ContractError> {
        if amount.is_zero() {
            return Err(ContractError::InvalidZeroAmount {});
        }

        let mut token_info = TOKEN_INFO.load(deps.storage)?;

        // Only the designated minter can mint
        let mint_data = token_info.mint.as_ref().ok_or(ContractError::Unauthorized {
            reason: "No minter configured — minting is disabled".to_string(),
        })?;

        if info.sender != mint_data.minter {
            return Err(ContractError::Unauthorized {
                reason: format!(
                    "{} is not the minter (minter is {})",
                    info.sender, mint_data.minter
                ),
            });
        }

        // Check cap
        let new_supply = token_info.total_supply + amount;
        if let Some(cap) = mint_data.cap {
            if new_supply > cap {
                return Err(ContractError::CapExceeded {
                    cap,
                    new_supply,
                });
            }
        }

        // Update total supply
        token_info.total_supply = new_supply;
        TOKEN_INFO.save(deps.storage, &token_info)?;

        // Credit recipient
        let rcpt_addr = deps.api.addr_validate(&recipient)?;
        BALANCES.update(
            deps.storage,
            &rcpt_addr,
            |balance| -> StdResult<_> {
                Ok(balance.unwrap_or_default() + amount)
            },
        )?;

        Ok(Response::new()
            .add_attribute("action", "mint")
            .add_attribute("to", rcpt_addr)
            .add_attribute("amount", amount))
    }
```

## 11. Execute — Send (Contract-to-Contract Transfers)

`Send` is what makes CW-20 tokens composable. It transfers tokens to a **contract** and sends that contract a `Cw20ReceiveMsg` so it can react to the incoming tokens. Think of it as a transfer + notification in a single message.

This is the CW-20 answer to ERC-20's `approve` + `transferFrom` — but safer and simpler. Instead of granting unlimited spending rights to a third party, you send tokens directly and include instructions for what the receiving contract should do.

Replace the `send` stub in the `exec` module:

```rust
    pub fn send(
        deps: DepsMut,
        info: MessageInfo,
        contract: String,
        amount: Uint128,
        msg: Binary,
    ) -> Result<Response, ContractError> {
        use cosmwasm_std::WasmMsg;
        use crate::msg::Cw20ReceiveMsg;

        if amount.is_zero() {
            return Err(ContractError::InvalidZeroAmount {});
        }

        let rcpt_addr = deps.api.addr_validate(&contract)?;

        // Deduct from sender (same logic as transfer)
        BALANCES.update(
            deps.storage,
            &info.sender,
            |balance| -> Result<_, ContractError> {
                let balance = balance.unwrap_or_default();
                if balance < amount {
                    return Err(ContractError::InsufficientFunds {
                        address: info.sender.clone(),
                        available: balance,
                        required: amount,
                    });
                }
                Ok(balance - amount)
            },
        )?;

        // Credit to the contract
        BALANCES.update(
            deps.storage,
            &rcpt_addr,
            |balance| -> StdResult<_> {
                Ok(balance.unwrap_or_default() + amount)
            },
        )?;

        // Build the Cw20ReceiveMsg that the receiving contract will handle
        let receive_msg = Cw20ReceiveMsg {
            sender: info.sender.to_string(),
            amount,
            msg,
        };

        // Create a WasmMsg to execute the receiving contract's Receive handler
        let exec_msg = WasmMsg::Execute {
            contract_addr: rcpt_addr.to_string(),
            msg: to_json_binary(&receive_msg)?,
            funds: vec![],
        };

        Ok(Response::new()
            .add_message(exec_msg)
            .add_attribute("action", "send")
            .add_attribute("from", info.sender)
            .add_attribute("to", rcpt_addr)
            .add_attribute("amount", amount))
    }
```

**How does this work with the Actor Model?**

Remember from Lab 3: when your contract finishes processing a message, it returns a `Response`. That response can contain **sub-messages** — instructions for the runtime to execute *after* your contract has completed. The `add_message(exec_msg)` line adds a `WasmMsg::Execute` to the response, telling the runtime: "after I'm done, call this other contract with this message."

The receiving contract sees a `Cw20ReceiveMsg` where `info.sender` is the **token contract's address** (not the original user). The `sender` field inside the message tells the receiving contract who originally sent the tokens. This way, the receiving contract can verify which token contract is calling it and prevent spoofing.

## 15. Exercises & Wrap-Up

### What We Covered

- **Token standards** — why they exist (composability), ERC vs CW naming, key differences between ERC-20 and CW-20.
- **CW-20 spec anatomy** — base (Transfer, Burn, Send) + extensions (Mintable, Allowances, Enumerable, Marketing).
- **State design** — `Item` for singleton metadata, `Map` for per-address balances, and why `Map` outperforms `Item<Vec<...>>` at scale.
- **Incremental build** — instantiate → queries → transfer → burn → mint → send, testing at each step.
- **The `Send` / `Receive` pattern** — how CW-20 replaces ERC-20's `approve` + `transferFrom` with a safer single-message flow.
- **Deployment** — optimized Docker build, instantiate with initial balances, real on-chain interactions.

### Optional Exercises (Take-Home)

1. **Allowances** — Implement the allowance extension: `IncreaseAllowance { spender, amount, expires }`, `DecreaseAllowance { spender, amount, expires }`, `TransferFrom { owner, recipient, amount }`, and `BurnFrom { owner, amount }`. You'll need a new `Map<(&Addr, &Addr), AllowanceInfo>` in state to track (owner, spender) → allowance pairs. The `cw-utils` crate provides an `Expiration` type you can use for the `expires` field.

2. **Build a Receiver Contract** — Create a second contract that implements the `Receive` handler. For example, a simple "token vault" that accepts CW-20 tokens via `Send` and lets the original sender withdraw them later. Deploy both contracts and test the `Send` → `Receive` flow on testnet.

3. **Enumerable Queries** — Add `AllAccounts { start_after, limit }` which returns a paginated list of all addresses that hold tokens. This requires using `BALANCES.range()` with the `Order::Ascending` option. Think about why pagination is important (hint: what if there are 100,000 holders?).

4. **Events & Attributes** — Our handlers already emit basic attributes. Add richer events using `Response::add_event()` with structured attributes (e.g., an event named `"cw20"` with key-value pairs). Query these events on the testnet explorer after executing transactions.

5. **Compare with cw20-base** — Clone the [cw-plus repository](https://github.com/CosmWasm/cw-plus) and explore `contracts/cw20-base/`. Compare the reference implementation with what you built today. What did they add that we skipped? How do they structure their code differently?

## Quick Reference — CW20 Messages We Implemented

| Message | Type | Who Can Call | What Changes |
|---------|------|-------------|-------------|
| `Transfer { recipient, amount }` | Execute | Anyone with balance | Sender ↓, Recipient ↑ |
| `Burn { amount }` | Execute | Anyone with balance | Sender ↓, Total Supply ↓ |
| `Mint { recipient, amount }` | Execute | Minter only | Recipient ↑, Total Supply ↑ |
| `Send { contract, amount, msg }` | Execute | Anyone with balance | Sender ↓, Contract ↑, triggers Receive |
| `Balance { address }` | Query | Anyone | — |
| `TokenInfo {}` | Query | Anyone | — |
| `Minter {}` | Query | Anyone | — |
