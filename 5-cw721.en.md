## 1. NFTs — What Makes a Token Non-Fungible?

### Fungible vs Non-Fungible

In Lab 4, we built a CW20 token. Every unit of that token was identical — your 1 LAB4 and my 1 LAB4 are interchangeable, just like two €1 coins. This property is called **fungibility**.

Non-fungible tokens are the opposite. Each token is **unique** — it has its own identity, its own owner, and its own metadata. Think of it like a concert ticket: even if two tickets cost the same, seat A12 is not the same as seat B7.

### What Changes at the Data Model Level?

This difference has profound implications for how we design storage:

| | CW20 (Fungible) | CW721 (Non-Fungible) |
|---|---|---|
| **Identity** | No individual token identity — just balances | Each token has a unique `token_id` (string) |
| **Storage** | `Map<&Addr, Uint128>` — address → balance | `Map<&str, TokenInfo>` — token_id → owner + metadata |
| **Transfer** | Subtract from sender, add to recipient | Change the `owner` field on one specific token |
| **Total supply** | A single `Uint128` counter | A count of minted tokens (or enumerate them) |
| **Metadata** | Shared (name, symbol, decimals) | Per-token (name, description, image, attributes) |

### ERC-721 (Ethereum) → CW-721 (CosmWasm)

Just like CW-20 mirrors ERC-20, CW-721 mirrors ERC-721 with CosmWasm-specific improvements:

1. **No reentrancy risk** — same Actor Model protection as CW-20.
2. **`SendNft` replaces `safeTransferFrom`** — Ethereum's ERC-721 has `safeTransferFrom` which calls `onERC721Received` on the recipient. CW-721's `SendNft` does the same thing: transfers the token and sends a `ReceiveNft` message to the receiving contract.
3. **Operators (`ApproveAll`)** — CW-721 supports both per-token approvals (like ERC-721's `approve`) and blanket operator approvals (like ERC-721's `setApprovalForAll`).
4. **On-chain metadata extensions** — ERC-721 typically stores a `tokenURI` pointing to off-chain JSON. CW-721 allows you to store arbitrary structured data directly on-chain via a generic extension type `T`.

### Real-World NFT Use Cases

NFTs aren't just profile pictures. They're used for:
- **Digital art & collectibles** — the most visible use case
- **Gaming items** — swords, armor, land parcels with unique stats
- **Event tickets** — provably unique, transferable, verifiable
- **Domain names** — ENS on Ethereum, Stargaze Names on Cosmos
- **Real-world asset certificates** — property deeds, academic credentials

In all of these cases, the key property is the same: each token is unique and individually owned.

## 2. The CW721 Specification

### Base (Required)

**Execute messages:**

| Message | What it does |
|---------|-------------|
| `TransferNft { recipient, token_id }` | Move ownership of a specific token to a wallet address |
| `SendNft { contract, token_id, msg }` | Transfer token to a contract and trigger `ReceiveNft` on it |
| `Approve { spender, token_id, expires }` | Grant one address permission to transfer one specific token |
| `Revoke { spender, token_id }` | Remove a previously granted per-token approval |
| `ApproveAll { operator, expires }` | Grant one address permission to transfer *all* your tokens |
| `RevokeAll { operator }` | Remove a blanket operator approval |

**Queries:**

| Query | What it returns |
|-------|----------------|
| `OwnerOf { token_id }` | Who owns this token + list of approvals |
| `NftInfo { token_id }` | The token's metadata (URI and/or on-chain extension) |
| `AllNftInfo { token_id }` | Combined `OwnerOf` + `NftInfo` (optimization for frontends) |
| `ContractInfo {}` | Collection name and symbol |
| `NumTokens {}` | Total number of minted tokens |

### Extensions

| Extension | What it adds |
|-----------|-------------|
| **Enumerable** | `Tokens { owner }`, `AllTokens {}` — paginated listing of token IDs |
| **Metadata on-chain** | Custom data stored directly in the token (not just a URI) |

### What We'll Build Today

We'll implement:
- **Instantiate** — set collection name/symbol and minter
- **Mint** — create new tokens with metadata
- **TransferNft** — change ownership
- **Approve / Revoke** — per-token approvals
- **SendNft** — transfer + notify a receiving contract
- **Queries** — OwnerOf, NftInfo, AllNftInfo, ContractInfo, NumTokens, AllTokens

We'll also add **on-chain metadata** — each NFT will carry structured data (name, description, image URL, custom attributes) directly in contract storage.

## 4. State Design — Thinking in Unique Tokens

The biggest conceptual shift from CW20 to CW721 is how we model storage. In CW20, every address had a single number (its balance). In CW721, every *token* is its own entity with an owner, metadata, and a list of approvals.

### 4.1 What We Need to Store

1. **Collection info** — the collection's name and symbol (singleton, like CW20's `TokenInfo`)
2. **Minter** — who is allowed to mint new tokens
3. **Token count** — how many tokens exist
4. **Token data** — for each token_id: owner, metadata, approvals
5. **Operator approvals** — blanket "this address can transfer all my NFTs" grants

### 4.2 Choosing Storage Primitives

| Data | Primitive | Key | Why |
|------|-----------|-----|-----|
| Collection info | `Item<CollectionInfo>` | `"collection_info"` | One per contract |
| Minter | `Item<Addr>` | `"minter"` | One per contract |
| Token count | `Item<u64>` | `"num_tokens"` | One per contract |
| Token data | `Map<&str, TokenData>` | `"tokens"` | One entry per token_id |
| Operators | `Map<(&Addr, &Addr), Expiration>` | `"operators"` | (owner, operator) → expiry |

> **Why `Map<&str, TokenData>` for tokens?** Token IDs are strings in CW721 (not numbers!). This is different from ERC-721 where token IDs are `uint256`. Strings give you flexibility: you can use UUIDs, sequential numbers, human-readable names, or any other scheme. The `Map` gives O(1) lookup by token_id.

> **Why a separate `Map` for operators?** Operator approvals are per-owner, not per-token. Alice can say "Bob can transfer any of my NFTs." This is a relationship between two addresses, not tied to a specific token — so it needs its own storage namespace. The composite key `(&Addr, &Addr)` maps (owner, operator) to an expiration.

### 4.3 Implement the State

Replace `src/state.rs`:

```rust
use cosmwasm_schema::cw_serde;
use cosmwasm_std::Addr;
use cw_storage_plus::{Item, Map};
use cw_utils::Expiration;

#[cw_serde]
pub struct CollectionInfo {
    pub name: String,
    pub symbol: String,
}

/// Per-token approval: "this spender can transfer this specific token"
#[cw_serde]
pub struct Approval {
    pub spender: Addr,
    pub expires: Expiration,
}

/// On-chain metadata stored with each NFT.
/// This is our custom extension — the CW721 spec allows any data here.
#[cw_serde]
pub struct Metadata {
    pub name: Option<String>,
    pub description: Option<String>,
    pub image: Option<String>,
}

/// Everything stored per token.
#[cw_serde]
pub struct TokenData {
    pub owner: Addr,
    pub token_uri: Option<String>,
    pub extension: Metadata,
    pub approvals: Vec<Approval>,
}

pub const COLLECTION_INFO: Item<CollectionInfo> = Item::new("collection_info");
pub const MINTER: Item<Addr> = Item::new("minter");
pub const NUM_TOKENS: Item<u64> = Item::new("num_tokens");
pub const TOKENS: Map<&str, TokenData> = Map::new("tokens");
pub const OPERATORS: Map<(&Addr, &Addr), Expiration> = Map::new("operators");
```

**Key design decisions:**

- **`Metadata` as a separate struct** — The CW721 spec uses a generic extension `T` for on-chain metadata. We're making our concrete: each token has an optional name, description, and image URL. In production, you'd make this generic. For learning, concrete types are clearer.

- **`approvals: Vec<Approval>` inside `TokenData`** — Per-token approvals are stored with the token itself. This is efficient because you always check approvals when processing a transfer, so the data is co-located. The vector is typically very short (0–3 entries).

- **`token_uri: Option<String>`** — The traditional pointer to off-chain JSON metadata (following the ERC-721 Metadata JSON Schema). Our contract supports *both* off-chain URI and on-chain extension.

Unlike CW20 where instantiation distributes initial token balances, CW721 instantiation is simpler: set the collection name/symbol and designate a minter. No tokens exist yet — they'll be minted later.

```rust
pub struct InstantiateMsg {

    pub name: String,
    /// Collection symbol (e.g. "CK", "DADN")
    pub symbol: String,

    #[returns(ContractInfoResponse)]

    /// Returns combined owner + metadata. Returns AllNftInfoResponse.
    #[returns(AllNftInfoResponse)]
    AllNftInfo { token_id: String },
}

// --- Responses ---

#[cw_serde]
pub struct ContractInfoResponse {
    pub name: String,
    pub symbol: String,
}

#[cw_serde]
pub struct NumTokensResponse {
    pub count: u64,
}

#[cw_serde]
pub struct NftInfoResponse {
    pub token_uri: Option<String>,
    pub extension: Metadata,
}

#[cw_serde]
pub struct OwnerOfResponse {
    pub owner: String,
    pub approvals: Vec<ApprovalResponse>,
}

#[cw_serde]
pub struct ApprovalResponse {
    pub spender: String,
    pub expires: Expiration,
}

    pub access: OwnerOfResponse,
    pub info: NftInfoResponse,
}
```

> **`AllNftInfo` is an optimization.** Frontends displaying an NFT need both ownership data and metadata. Without this combined query, they'd need two separate queries per token. `AllNftInfo` bundles both in one round trip.

```rust
    to_json_binary, Binary, Deps, DepsMut, Env, MessageInfo, Response, StdResult,

use crate::error::ContractError;
use crate::msg::{InstantiateMsg, QueryMsg};
use crate::state::{CollectionInfo, COLLECTION_INFO, MINTER, NUM_TOKENS};

pub fn instantiate(
    deps: DepsMut,
    _env: Env,
    _info: MessageInfo,
    msg: InstantiateMsg,

    if msg.name.is_empty() || msg.symbol.is_empty() {
        return Err(ContractError::Std(
            cosmwasm_std::StdError::generic_err("Name and symbol must not be empty"),

    MINTER.save(deps.storage, &minter)?;
    NUM_TOKENS.save(deps.storage, &0u64)?;

    Ok(Response::new()
        .add_attribute("action", "instantiate")
        .add_attribute("minter", minter))
}

    unimplemented!()
}
```

## 6. Queries — ContractInfo & NftInfo

### 6.1 Implement the Query Handlers

Replace the `query` function in `src/contract.rs` and add the query module:

```rust
pub fn query(deps: Deps, env: Env, msg: QueryMsg) -> StdResult<Binary> {
    use QueryMsg::*;

    match msg {
        ContractInfo {} => to_json_binary(&query::contract_info(deps)?),
        NumTokens {} => to_json_binary(&query::num_tokens(deps)?),
        NftInfo { token_id } => to_json_binary(&query::nft_info(deps, token_id)?),
        OwnerOf { token_id } => to_json_binary(&query::owner_of(deps, env, token_id)?),
        AllNftInfo { token_id } => to_json_binary(&query::all_nft_info(deps, env, token_id)?),
    }
}

mod query {
    use super::*;
    use crate::msg::{
        AllNftInfoResponse, ApprovalResponse, ContractInfoResponse, NftInfoResponse,
        NumTokensResponse, OwnerOfResponse,
    };
    use crate::state::TOKENS;

    pub fn contract_info(deps: Deps) -> StdResult<ContractInfoResponse> {
        let info = COLLECTION_INFO.load(deps.storage)?;
        Ok(ContractInfoResponse {
            name: info.name,
            symbol: info.symbol,
        })
    }

    pub fn num_tokens(deps: Deps) -> StdResult<NumTokensResponse> {
        let count = NUM_TOKENS.load(deps.storage)?;
        Ok(NumTokensResponse { count })
    }

    pub fn nft_info(deps: Deps, token_id: String) -> StdResult<NftInfoResponse> {
        let token = TOKENS.load(deps.storage, &token_id)?;
        Ok(NftInfoResponse {
            token_uri: token.token_uri,
            extension: token.extension,
        })
    }

    pub fn owner_of(deps: Deps, env: Env, token_id: String) -> StdResult<OwnerOfResponse> {
        let token = TOKENS.load(deps.storage, &token_id)?;

        // Filter out expired approvals
        let approvals: Vec<ApprovalResponse> = token
            .approvals
            .into_iter()
            .filter(|a| !a.expires.is_expired(&env.block))
            .map(|a| ApprovalResponse {
                spender: a.spender.to_string(),
                expires: a.expires,
            })
            .collect();

        Ok(OwnerOfResponse {
            owner: token.owner.to_string(),
            approvals,
        })
    }

    pub fn all_nft_info(deps: Deps, env: Env, token_id: String) -> StdResult<AllNftInfoResponse> {
        let access = owner_of(deps, env, token_id.clone())?;
        let info = nft_info(deps, token_id)?;
        Ok(AllNftInfoResponse { access, info })
    }
}
```

**Notice the approval filtering in `owner_of`.** Approvals have expiration times. Rather than eagerly cleaning up expired approvals (which would cost gas on every interaction), we filter them lazily at query time. This is a common pattern in CosmWasm: let stale data sit in storage and filter it when reading.

## 8. Execute — Mint

Minting is how NFTs come into existence. Only the designated minter can create new tokens. Each mint call creates exactly one token with a unique ID.

### 8.1 Add Mint to ExecuteMsg

Replace the empty `ExecuteMsg` in `src/msg.rs`:

```rust
// --- Execute ---

#[cw_serde]
pub enum ExecuteMsg {
    /// Create a new NFT. Only callable by the minter.
    Mint {
        token_id: String,
        owner: String,
        token_uri: Option<String>,
        extension: Metadata,
    },
    /// Transfer ownership of a token to a wallet address.
    TransferNft {
        recipient: String,
        token_id: String,
    },
    /// Grant permission to transfer a specific token.
    Approve {
        spender: String,
        token_id: String,
        expires: Option<Expiration>,
    },
    /// Remove a per-token approval.
    Revoke {
        spender: String,
        token_id: String,
    },
    /// Transfer token to a contract and trigger ReceiveNft.
    SendNft {
        contract: String,
        token_id: String,
        msg: Binary,
    },
}

/// The message a contract receives when an NFT is `SendNft`-ed to it.
#[cw_serde]
pub struct Cw721ReceiveMsg {
    pub sender: String,
    pub token_id: String,
    pub msg: Binary,
}
```

> **Compare with CW20's `Mint`.** CW20 minting takes a `recipient` and `amount`. CW721 minting takes a `token_id`, `owner`, and the full metadata payload. Each mint creates exactly one unique token. There is no concept of minting "5 NFTs" in a single message — each one is an individual call with its own ID and metadata.

### 8.2 Implement Mint

Add the execute dispatch and the mint handler in `src/contract.rs`. Replace the stub `execute` function:

```rust
use crate::msg::ExecuteMsg;

pub fn execute(
    deps: DepsMut,
    env: Env,
    info: MessageInfo,
    msg: ExecuteMsg,
) -> Result<Response, ContractError> {
    use ExecuteMsg::*;

    match msg {
        Mint {
            token_id,
            owner,
            token_uri,
            extension,
        } => exec::mint(deps, info, token_id, owner, token_uri, extension),
        TransferNft {
            recipient,
            token_id,
        } => exec::transfer_nft(deps, env, info, recipient, token_id),
        Approve {
            spender,
            token_id,
            expires,
        } => exec::approve(deps, env, info, spender, token_id, expires),
        Revoke { spender, token_id } => exec::revoke(deps, info, spender, token_id),
        SendNft {
            contract,
            token_id,
            msg: send_msg,
        } => exec::send_nft(deps, env, info, contract, token_id, send_msg),
    }
}

mod exec {
    use super::*;
    use crate::state::{Metadata, TokenData, MINTER, TOKENS};

    pub fn mint(
        deps: DepsMut,
        info: MessageInfo,
        token_id: String,
        owner: String,
        token_uri: Option<String>,
        extension: Metadata,
    ) -> Result<Response, ContractError> {
        // Only the minter can mint
        let minter = MINTER.load(deps.storage)?;
        if info.sender != minter {
            return Err(ContractError::Unauthorized {
                reason: format!("{} is not the minter", info.sender),
            });
        }

        // Token ID must not already exist
        if TOKENS.has(deps.storage, &token_id) {
            return Err(ContractError::TokenAlreadyExists {
                token_id: token_id.clone(),
            });
        }

        // Validate owner address
        let owner_addr = deps.api.addr_validate(&owner)?;

        // Store the token
        let token = TokenData {
            owner: owner_addr.clone(),
            token_uri,
            extension,
            approvals: vec![],
        };
        TOKENS.save(deps.storage, &token_id, &token)?;

        // Increment token count
        NUM_TOKENS.update(deps.storage, |count| -> StdResult<u64> {
            Ok(count + 1)
        })?;

        Ok(Response::new()
            .add_attribute("action", "mint")
            .add_attribute("token_id", &token_id)
            .add_attribute("owner", owner_addr))
    }
```

## 9. Execute — TransferNft

Transfer changes ownership of a specific token. Only the owner, an approved spender, or an operator can transfer.

### 9.1 Authorization Helper

Before implementing transfer, we need a helper function that checks whether `info.sender` is authorized to move a given token. This logic is shared between `TransferNft`, `SendNft`, and later `Approve`/`Revoke`.

Add this helper inside the `exec` module:

```rust
    /// Check if `sender` is authorized to transfer `token_id`.
    /// Returns the token data if authorized, or an error.
    fn check_can_send(
        deps: &DepsMut,
        env: &Env,
        sender: &Addr,
        token_id: &str,
    ) -> Result<TokenData, ContractError> {
        let token = TOKENS
            .may_load(deps.storage, token_id)?
            .ok_or_else(|| ContractError::TokenNotFound {
                token_id: token_id.to_string(),
            })?;

        // Owner can always transfer
        if token.owner == *sender {
            return Ok(token);
        }

        // Check per-token approvals
        let has_approval = token
            .approvals
            .iter()
            .any(|a| a.spender == *sender && !a.expires.is_expired(&env.block));
        if has_approval {
            return Ok(token);
        }

        // Check operator approvals (blanket approval for all owner's tokens)
        let is_operator = OPERATORS
            .may_load(deps.storage, (&token.owner, sender))?
            .map(|exp| !exp.is_expired(&env.block))
            .unwrap_or(false);
        if is_operator {
            return Ok(token);
        }

        Err(ContractError::Unauthorized {
            reason: format!("{} is not authorized to transfer token {}", sender, token_id),
        })
    }
```

**The authorization cascade:** owner → per-token approval → operator. If any of these pass, the sender is authorized. This mirrors ERC-721's `_isApprovedOrOwner` check.

```rust
    use cosmwasm_std::Order;
    use crate::msg::TokensResponse;

    const DEFAULT_LIMIT: u32 = 10;
    const MAX_LIMIT: u32 = 30;

    pub fn all_tokens(
        deps: Deps,
        start_after: Option<String>,
        limit: Option<u32>,
    ) -> StdResult<TokensResponse> {
        let limit = limit.unwrap_or(DEFAULT_LIMIT).min(MAX_LIMIT) as usize;

        let start = start_after.as_deref().map(cosmwasm_std::Bound::exclusive);

        let tokens: Vec<String> = TOKENS
            .keys(deps.storage, start, None, Order::Ascending)
            .take(limit)
            .collect::<StdResult<_>>()?;

        Ok(TokensResponse { tokens })
    }

    pub fn tokens(
        deps: Deps,
        owner: String,
        start_after: Option<String>,
        limit: Option<u32>,
    ) -> StdResult<TokensResponse> {
        let limit = limit.unwrap_or(DEFAULT_LIMIT).min(MAX_LIMIT) as usize;
        let owner_addr = deps.api.addr_validate(&owner)?;

        let start = start_after.as_deref().map(cosmwasm_std::Bound::exclusive);

        // Iterate all tokens and filter by owner
        // (In production, you'd use a secondary index for efficiency)
        let tokens: Vec<String> = TOKENS
            .range(deps.storage, start, None, Order::Ascending)
            .filter_map(|item| {
                item.ok().and_then(|(id, data)| {
                    if data.owner == owner_addr {
                        Some(id)
                    } else {
                        None
                    }
                })
            })
            .take(limit)
            .collect();

        Ok(TokensResponse { tokens })
    }
```

> **Why `start_after` / `limit` instead of page numbers?** Page-based pagination breaks when items are added or removed between pages. Cursor-based pagination (using `start_after`) is stable: "give me the next 10 tokens after token_id X" always returns the correct results, even if tokens were minted or burned between calls.

> **Performance note on `tokens()` (owner filter):** Our implementation iterates all tokens and filters by owner. This works fine for small collections (hundreds of tokens), but for collections with millions of tokens, you'd want a secondary index: a `Map<(&Addr, &str), Empty>` keyed by `(owner, token_id)` that lets you iterate directly. Adding that index is left as an exercise.

## 13. Final Test Suite

Let's add a comprehensive lifecycle test that exercises the full flow:

```rust
    #[test]
    fn full_lifecycle() {
        let mut app = App::default();
        let msg = default_instantiate_msg(&app);
        let contract_addr = setup_contract(&mut app, msg);

        let minter = app.api().addr_make("minter");
        let alice = app.api().addr_make("alice");
        let bob = app.api().addr_make("bob");
        let charlie = app.api().addr_make("charlie");

        // 1. Mint two tokens to Alice
        mint_token(&mut app, &contract_addr, &minter, "dragon", &alice);
        mint_token(&mut app, &contract_addr, &minter, "phoenix", &alice);

        let count: NumTokensResponse = app
            .wrap()
            .query_wasm_smart(&contract_addr, &QueryMsg::NumTokens {})
            .unwrap();
        assert_eq!(count.count, 2);

        // 2. Alice transfers "dragon" to Bob
        app.execute_contract(
            alice.clone(),
            contract_addr.clone(),
            &ExecuteMsg::TransferNft {
                recipient: bob.to_string(),
                token_id: "dragon".to_string(),
            },
            &[],
        )
        .unwrap();

        // 3. Alice approves Charlie for "phoenix"
        app.execute_contract(
            alice.clone(),
            contract_addr.clone(),
            &ExecuteMsg::Approve {
                spender: charlie.to_string(),
                token_id: "phoenix".to_string(),
                expires: None,
            },
            &[],
        )
        .unwrap();

        // 4. Charlie transfers "phoenix" to Bob using the approval
        app.execute_contract(
            charlie.clone(),
            contract_addr.clone(),
            &ExecuteMsg::TransferNft {
                recipient: bob.to_string(),
                token_id: "phoenix".to_string(),
            },
            &[],
        )
        .unwrap();

        // --- Verify final state ---
        // Bob owns both tokens
        let resp: TokensResponse = app
            .wrap()
            .query_wasm_smart(
                &contract_addr,
                &QueryMsg::Tokens {
                    owner: bob.to_string(),
                    start_after: None,
                    limit: None,
                },
            )
            .unwrap();
        assert_eq!(resp.tokens, vec!["dragon", "phoenix"]);

        // Alice owns nothing
        let resp: TokensResponse = app
            .wrap()
            .query_wasm_smart(
                &contract_addr,
                &QueryMsg::Tokens {
                    owner: alice.to_string(),
                    start_after: None,
                    limit: None,
                },
            )
            .unwrap();
        assert!(resp.tokens.is_empty());

        // Token count unchanged (no burning)
        let count: NumTokensResponse = app
            .wrap()
            .query_wasm_smart(&contract_addr, &QueryMsg::NumTokens {})
            .unwrap();
        assert_eq!(count.count, 2);
    }
```

## 16. Exercises & Wrap-Up

### What We Covered

- **NFTs vs Fungible tokens** — unique identity per token, ownership instead of balances, per-token metadata.
- **CW-721 spec** — base messages (Transfer, Send, Approve, Revoke) + queries (OwnerOf, NftInfo, AllNftInfo, ContractInfo, NumTokens).
- **State design** — `Map<&str, TokenData>` for tokens, `Map<(&Addr, &Addr), Expiration>` for operators, composite keys in `cw-storage-plus`.
- **Authorization cascade** — owner → per-token approval → operator, with expiration checks.
- **On-chain metadata** — custom `Metadata` struct stored alongside each token.
- **Paginated enumeration** — `start_after` / `limit` cursor-based pagination with `Map::keys()` and `Map::range()`.
- **Approval lifecycle** — grant, use, auto-clear on transfer, revoke.

### Optional Exercises (Take-Home)

1. **Burn** — Add a `Burn { token_id }` execute message that lets the owner destroy a token. Remember to decrement `NUM_TOKENS` and remove the entry from `TOKENS`. Who should be able to burn — only the owner, or also approved spenders?

2. **ApproveAll / RevokeAll** — Implement operator approvals. `ApproveAll { operator, expires }` grants an address the right to transfer *any* token owned by the sender. Use the `OPERATORS` map we defined in state. Add a query `ApprovedForAll { owner }` that returns a list of operators for a given owner.

3. **Owner Index** — Add a secondary index `Map<(&Addr, &str), Empty>` that maps `(owner, token_id)` to allow efficient lookup of tokens by owner without scanning the full `TOKENS` map. Update `mint`, `transfer_nft`, and `send_nft` to maintain this index. Then rewrite the `tokens()` query to use it.

4. **Expiration Testing** — Write a test that approves a spender with `Expiration::AtHeight(100)`, advances the block height using `app.update_block(|b| b.height = 101)`, and verifies that the approval is no longer valid. This exercises the time-based security of the approval system.

5. **Compare with cw721-base** — Clone the [cw-nfts repository](https://github.com/public-awesome/cw-nfts) and explore `contracts/cw721-base/`. Compare with what you built. Notable differences: the reference implementation uses Rust generics for the extension type `T`, supports `UpdateOwnership` for minter transfer, and has a much more extensive test suite.

## Quick Reference — CW721 Messages We Implemented

| Message | Type | Who Can Call | What Changes |
|---------|------|-------------|-------------|
| `Mint { token_id, owner, ... }` | Execute | Minter only | New token created, NUM_TOKENS ↑ |
| `TransferNft { recipient, token_id }` | Execute | Owner / Approved / Operator | Token owner changes, approvals cleared |
| `Approve { spender, token_id, expires }` | Execute | Owner only | Approval added to token |
| `Revoke { spender, token_id }` | Execute | Owner only | Approval removed from token |
| `SendNft { contract, token_id, msg }` | Execute | Owner / Approved / Operator | Owner changes, triggers ReceiveNft |
| `ContractInfo {}` | Query | Anyone | — |
| `NumTokens {}` | Query | Anyone | — |
| `NftInfo { token_id }` | Query | Anyone | — |
| `OwnerOf { token_id }` | Query | Anyone | — |
| `AllNftInfo { token_id }` | Query | Anyone | — |
| `AllTokens { start_after, limit }` | Query | Anyone | — |
| `Tokens { owner, start_after, limit }` | Query | Anyone | — |
